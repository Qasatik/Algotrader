Статус после первого успешного запуска на mainnet (2026-07-15):

## P0 — Критическая безопасность (до масштабирования капитала)

### 1. Реконсиляция позиции при старте  ✅ (commit ee8ad78)
**Проблема:** [`CarryStrategy.__init__`](core/carry_strategy.py:90) жёстко
задаёт `state = CarryState.FLAT`. Если бот упал / перезагрузился / закрылся
терминал, а позиция открыта — при рестарте он думает, что он «плоский», и
попытается **открыть ВТОРУЮ позицию поверх существующей**.

**Решение:** При старте вызывать `exchange.get_positions(symbol)` и читать
реальный размер перпа + спот-баланс. Если хедж-пара существует — выставить
`state = HEDGED`, `position_qty = <факт>`, `entry_basis_bps = <текущий>`.
Если есть только одна нога (осиротевшая) — закрыть её перед торговлей.

**Тесты:** `test_reconciles_open_position_on_startup`,
`test_flattens_orphaned_leg_on_startup`.

### 2. Реализовать настоящую ребалансировку  ✅ (commit 92baef1)
**Проблема:** [`_rebalance()`](core/carry_strategy.py:305) — **заглушка,
которая ничего не делает**: логирует «rebalance» и возвращает
`{"rebalanced": True}` без единой заявки. Когда базис уплывает >20bps,
стратегия *говорит*, что ребалансирует, но хедж остаётся рассинхронизированным.
Со временем перп и спот расходятся (комиссии, частичные исполнения, фандинг),
и позиция перестаёт быть дельта-нейтральной.

**Решение:** Считать реальную дельту между размером шорт-перпа и лонг-спота
(читать оба с биржи), затем подрезать/добавить меньшую ногу, чтобы
нотионалы совпадали. Помнить: спот Market BUY принимает qty в **USDT**,
SELL — в **BTC**.

**Тесты:** `test_rebalance_trims_oversized_leg`, `test_rebalance_noop_when_aligned`.

### 3. Разделить «остановку бота» и «закрытие позиции»  ✅ (commit 92baef1)
**Проблема:** [`run_carry_testnet.py:128`](scripts/run_carry_testnet.py:128)
автоматически закрывает позицию по SIGINT. Для кэрри-трейда обычно хочется
**продолжать собирать фандинг** между рестартами — закрытие на каждой
остановке выбрасывает позицию и платит комиссии туда-обратно.

**Решение:** Добавить флаг `--flatten-on-exit` (по умолчанию **выключен**).
При остановке — сохранить состояние и выйти; закрывать только если флаг
передан. Вместе с P0-1 (реконсиляцией) бот будет чисто возобновлять
открытую позицию.

## P1 — Надёжность для работы без присмотра

### 4. Запуск как systemd-сервис  ✅ (commit b330a04)
**Проблема:** Бот работает в терминале на переднем плане. Умирает при
перезагрузке, отключении SSH или случайном закрытии.

**Решение:** Написать `systemd`-юнит (`carry-bot.service`) с
`Restart=always`, `RestartSec=10`, `User=pepe`, логированием в journald.
`systemctl enable --now`. Переживает перезагрузки и авто-рестартует при падении.

### 5. Push-уведомления в Telegram о событиях  ✅ (commit 52f7581)
**Проблема:** Telegram сейчас **только по запросу** (`/status`, `/stats`,
`/trades`). Об открытии/закрытии/ошибке узнаёшь, только когда сам спросишь.

**Решение:** Подключить [`_broadcast()`](bot/telegram_admin.py:343) к циклу
кэрри. Пушить при: открытии позиции, закрытии (с причиной + P&L),
срабатывании basis-guard, откате при ошибке спота, повторных ошибках опроса.
Сообщения краткие (одна строка + эмодзи).

### 6. Heartbeat / контроль «жив»  ✅ (commit 52f7581)
**Проблема:** Если цикл завис или тихо умер — никто не заметит, пока не
проверишь вручную.

**Решение:** Каждые N опросов слать heartbeat в Telegram («жив, хеджирован,
фандинг=+0,0071%»). Лучше — watchdog, который алертит, если heartbeat не
пришёл за 2× интервала опроса (ловит зависший цикл).

### 7. Логирование в файл с ротацией  ✅ (commit 52f7581)
**Проблема:** Логи идут только в консоль — теряются при закрытии терминала.

**Решение:** Добавить `RotatingFileHandler` в [`utils/logger.py`](utils/logger.py),
пишущий в `logs/carry.log` (например 5МБ × 5 файлов). Консоль оставить тоже.

## P2 — Прибыльность и интеллект

### 8. Учёт реализованного фандинга + P&L  🟡
**Проблема:** CSV-лог пишет *действия* (open/close/rebalance), но не
*заработанные доллары*. [`carry_stats`](core/carry_stats.py) показывает
количества и средние, но никогда реальную прибыль.

**Решение:**
- При CLOSE забирать `get_closed_pnl()` для реализованного P&L перпа и
  записывать его.
- Отслеживать себестоимость спот-ноги vs выручки от продажи.
- Добавить колонку `funding_income_usdt`, заполняемую по дельтам баланса
  кошелька на таймстампах фандинга.
- Показывать `/pnl` в Telegram (сегодня / 7д / за всё время в $).

### 9. Сканер фандинга по нескольким символам  ✅
**Проблема:** Торгуется только BTCUSDT. На альтах фандинг часто выше
(ETH, SOL и т.д.).

**Решение:** Сканер, опрашивающий фандинг по вотчлисту, ранжирующий по
ставке и распределяющий капитал по топ-N символам (с учётом `max_notional`
на символ). Обобщить `CarryStrategy` на управление несколькими независимыми
парами.

**Готово:**
- [`scan_funding.py`](scripts/scan_funding.py) — сканер ставок по 14 символам
  с сортировкой по доходности и аннуализированной ставкой.
- [`run_carry_multi.py`](scripts/run_carry_multi.py) — мульти-символьный раннер:
  N стратегий одновременно, авто-определение `qty_step`, разделение капитала.
- `CarryConfig.base_coin` — свойство для определения базовой монеты (BTCUSDT→BTC).
- [`exchange.get_qty_step()`](core/exchange.py) — авто-определение шага лота.

### 10. Максимальное время удержания / выход при затухании фандинга  ✅
**Проблема:** Позиция может висеть бесконечно, даже когда фандинг почти ноль
(чуть выше `close_funding`), занимая капитал под копеечную доходность.

**Решение:** Добавить `max_hold_hours` (напр. 72ч) и порог
`min_yield_to_hold`, закрывающий позицию, если ожидаемый остаточный кэрри
ниже порога.

**Готово:** `CarryConfig.max_hold_hours` (0=без лимита). В `_decide_hedged()`
проверяется время удержания; при превышении — закрытие. CLI: `--max-hold-hours`.
Также: мониторинг ликвидации (`liq_warning_pct`) и исправлен баг в `_close()`
(продажа реального спот-баланса, а не размера перпа).

### 11. Мониторинг цены ликвидации + exchange-side SL  ✅ (commit b9a1a47)
**Проблема:** Basis-guard (50bps) *должен* поймать сквиз до ликвидации, но
явной проверки реальной цены лика нет.

**Решение:** Читать `liqPrice` из позиций; если цена к ней приближается
(напр. в пределах 15%) — принудительно закрыть + алерт. Запасная мера за
basis-guard.

## P3 — Усиление и операционка

### 12. Конфиг-файл (TOML) вместо аргументов CLI  ✅
Долгоиграющим сервисам не нужна командная строка из 9 аргументов.
[`config/loader.py`](config/loader.py:1) читает `config/carry.toml` (stdlib
`tomllib`) и подставляет значения через `ArgumentParser.set_defaults()`.
Приоритет: **CLI > TOML > дефолт**. Пример — [`config/carry.toml.example`](config/carry.toml.example:1).
Оба раннера (`run_carry_testnet.py`, `run_carry_multi.py`) поддерживают `--config`.

### 13. Аудит прав API-ключа  ✅
[`core/security.py`](core/security.py:1) при старте запрашивает скоуп ключа
через Bybit V5 `get_api_key_information` и проверяет: есть ли Trade-права,
и **НЕТ** ли опасных (`Wallet`/`AccountTransfer`/`Withdraw`). Блокирует запуск
на mainnet если ключ может двигать деньги; предупреждает если нет IP-whitelist
или ключ скоро истекает. Текущий ключ: `Spot+Derivatives Trade`, `Wallet=[]` —
✅ безопасен (выводов нет). Флаг `--skip-api-audit` для обхода. 11 тестов.

### 14. Логика реконнекта при длительных сбоях  ✅
В `_request` есть tenacity-retry, но долгий сбой Bybit просто спамил ошибки.
Добавлен экспоненциальный backoff: [`utils/backoff.py`](utils/backoff.py:1)
(`backoff_seconds()`) — первые 2 ошибки держат обычный каденс, дальше
`base × 2^(errors-grace)` с потолком 300с. Оба раннера используют его в sleep.
Telegram-алерт после 5 подряд неудач уже был в раннере.

### 15. Точность хеджа (выровнять qty после комиссий)  ✅
Параметр `spot_taker_fee` (0.1% по умолчанию) в [`CarryConfig`](core/carry_strategy.py:50).
При открытии спот-ордер gross-up на `1/(1-fee)`: вместо `qty×price` USDT
отправляется `qty×price/(1-fee)`, чтобы NET полученный BTC (после вычета
комиссии из base coin) точно равнялся шорт-перпу. Дельта остаётся по-настоящему
нейтральной с первого тика, без ожидания ребалансировки. 2 новых теста.

### 16. Симуляция заявок в dry-run  ✅
[`core/paper_trader.py`](core/paper_trader.py:1) — движок симулированных сделок.
В `--dry-run` режиме раннер прогоняет **живую** логику решений (`CarryStrategy`)
против реального фандинга/цены, но исполняет действия в виртуальном трекере:
начисляет фандинг каждые 8ч, списывает комиссии (perp0.055% + spot0.1%),
считает unrealised/realised P&L. Отчёт печатается на каждом heartbeat.
Позволяет увидеть реальную доходность стратегии без риска. 10 тестов.

## P4 — Мульти-символьная торговля и автоматизация

### 17. Мульти-символьный рантайм  ✅
[`run_carry_multi.py`](scripts/run_carry_multi.py:1) крутит N стратегий одновременно,
делит капитал поровну по слотам, авто-детектит `qty_step` (BTC=0.001, ETH=0.01 …)
через [`get_qty_step()`](core/exchange.py:227). Стратегия стала символ-агностичной
через свойство `base_coin` у [`CarryConfig`](core/carry_strategy.py:50).

### 18. Сканер фандинга  ✅
[`scan_funding.py`](scripts/scan_funding.py:1) ранжирует 14 мажорных пар по ставке
фандинга с годовой доходностью и базисом — быстрый поиск лучших carry-кандидатов.

### 19. Динамическая ротация символов (--top-N)  ✅
В `--top-n N` режиме рантайм каждые `--rebalance-cycles` опросов пересканирует
вселенную, ранжирует по фандингу и держит открытыми только top-N. Выпавшие из
топа символы продолжают мониториться (close/rebalance), но не открывают новые
позиции — параметр `can_open` в [`decide()`](core/carry_strategy.py:242).

### 20. Ежедневный автоматический code review  ✅
[`code_review.py`](scripts/code_review.py:1) — AST-сканер (секреты, длинные функции,
bare-except, TODO, типы). GitHub Actions cron
[`code-review.yml`](.github/workflows/code-review.yml:1) гоняет его ежедневно
в 12:00 МСК + ruff, отчёт падает в Job Summary.
